## 2. Missing Items Report

Nothing blocks approval. Two non-critical gaps are worth tracking:

1. **Circle-announcement read access is admin/creator-only, not full circle-membership-aware.**
   `AnnouncementsController` explicitly throws `ForbiddenException` for circle-scoped announcement reads by non-creator, non-admin callers, with an inline comment: *"Circle-member access will be available once membership is validated server-side."* This means a Sheikh/Supervisor/Student/Parent who is a legitimate circle member but not the announcement's creator currently cannot read that circle's announcements — likely narrower than intended, since circle announcements exist specifically to reach circle members. Needs circle-membership validation (via the existing Circles module) wired into the read path.
2. **Email delivery requires the caller to pre-supply `recipientEmail`.**
   Already captured as a proposed follow-up task ("Make sure notification emails still reach parents/students without extra manual work") — no `UsersModule` lookup exists yet to resolve a recipient's email from their `recipientId` automatically, so any call site that forgets to pass `notification.data.recipientEmail` gets a silent `FAILED` status instead of a delivered email.

No missing modules, controllers, guards, schemas, or permission wiring were found.

## 4. Technical Debt Report

1. **Circle-membership-aware announcement access** — see Missing Items §1. Should be resolved before circle announcements are relied on in production.
2. **Automatic email address resolution for notifications** — see Missing Items §2. Already tracked as follow-up task.
3. **No operator alerting on "email disabled" state** — `SmtpEmailProvider`'s silent-skip behavior is correct for dev, but production should surface a startup-time warning more prominently (e.g. a health-check flag) rather than only a log line.
4. **Thin-CRUD modules (Notification Templates, User Preferences) skip the use-case layer**, putting authorization/business logic directly in controllers. This is consistent with how the codebase already treats simple CRUD elsewhere, so it's a documented convention rather than a defect — flagged only for awareness if the team later wants use-cases everywhere for consistency.
5. Pre-existing, not Phase 10-specific: Mongoose duplicate-index warnings and npm audit vulnerabilities — both already captured as separate proposed follow-up tasks (#3 and #4) so they aren't duplicated here.

Phase 10 is solid: fully wired, RBAC- and tenant-isolated, no hardcoded secrets, compiles clean, and boots against a live database. **Proceed to prepare the Phase 11 architecture plan.** Implementation of Phase 11 should wait for explicit approval, per project convention.
